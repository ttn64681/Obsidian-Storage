# Autoencoding
- **What is it?:** multilayer neural network w/ target output = input
	- FLC or MultiLayer
- **Purpose:** trying to reconstruct original network
-  ![[Autoencoders.png]]
- **Goal:** Minimizing reconstruction error


- **Why** do we need to train network to reconstruct itself?
	- Generated output is not completely same:
		- **Denoise auto-encoder**:
			- have image with lots of noise or blur
			- reconstruct image and remove noise pixels

	- We are not exactly copying input
	- Side effect is compression effect
	- We can utilize this compression and split the network into 2 parts, and use them for:
		- e.g. YouTube provider: first compress video in company, then send it out to internet
			- client-side receives compressed video, and on their device it decompresses to get higher resolution once clicked/loaded?
			- **thus, reducing communication cost**

## Do we need labels for Autoencoding?
- Don't need labels for this
- we can just download clean images and manually add noise to the clean images
- then we train with noisy input images, and have the clean images as the output ground truth labels

## Regularizations on AutoEncoders
- **autoencoders need to be undercomplete:** 
- *the coded data (bottleneck layer) need to have a smaller dimension than input data:*
	- to avoid directly copying from the input to the output
	- to force feature learning and compression
		- if bottleneck layer is smaller, autoencoder is forced to learn most isgnificant features or a compressed representation of the input data

	- if you don't make it smaller, there's no way to compress
	- if you keep same weights, there's no way to compress, even if model is wider in the middle
	- thus we need to make it undercomplete

- Equivalently, we can add regularization on objective function
	- Loss Function
	- L(x, g(f(x))) + Omega(h)
	- more compressed representation of data

- We can use ReLU activation func to produce sparse encoded data
- In this way, the robustness and generalizability can be improved.

## Unsampling - deconvolution
- convolution vs deconvolution (transposed conv)
- Conv:
	- Cx = y
	- no padding, no strides
- DeConv:
	- C^T y = x'
	- no padding, no strides, transposed

- Deconv: reverse convolution
- **original conv:**
	- GMM format w/ input and weight, weight kernel * input matrix = output
- **deconv:**
	- Transposed GMM format w/ input and weight (Sparse C^T) * 
	- adding stride size creates sparse input feature map
	- when using stride, it might have downside effects (**checkerboard artifacts**)

## Checkerboard Artifacts:
- this is b/c when we include stride, and slide goes over original image, then some parts contain more information than other and overlap happens from the upscaling
	- To avoid checkerboard artifacts, **ensure kernel size is divisible by stride**, to ensure no odd amount. if even, no overlap
